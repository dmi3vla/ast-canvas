# Infinity Canvas — MVP brief для постановщика

## Зачем продукт

Infinity Canvas помогает быстро понять незнакомый репозиторий без прыжков между IDE, файлами и вкладками.

Пользователь открывает папку и сразу получает одну читаемую карту проекта. Слева остаётся карта, справа раскрывается смысл выбранной области, её зависимости и исходный код. Это **research/workspace для понимания кода**, не IDE и не генератор «идеальной» диаграммы.

## Визуальный контракт

Референс внешнего вида и плотности интерфейса: [`docs/proto1.png`](./proto1.png).

При принятии визуальных решений референс важнее текущего demo, случайного LLM-ответа и удобства реализации. Но не нужно копировать PNG попиксельно: нужно сохранить его характер — тёмный canvas, читаемые карточки, ясная иерархия, компактные детали и отсутствие IDE-шумов.

## Один главный пользовательский сценарий

```text
Open Folder
  → карта архитектурных областей на LEFT
  → click area
  → краткое описание на RIGHT
  → Codemap
  → реальные зависимости и связанные файлы
  → click file
  → исходник на RIGHT, LEFT остаётся видимым
```

Если этот путь не работает на небольшом fixture и на текущем репозитории, MVP не принят.

## MVP: что пользователь должен видеть

### 1. Старт

- Одно Electron-окно, без лишних welcome-экранов.
- Верхняя панель содержит: **Open Folder**, Regenerate, Demo только как вспомогательное действие.
- До выбора проекта: спокойный empty state с одним понятным CTA.
- После выбора проекта: canvas не заменяется экраном загрузки; показывается progress/status поверх или в toolbar.

### 2. Левая часть — карта

- LEFT занимает примерно 60% окна, размер регулируется splitter-ом.
- Карта всегда остаётся на месте при выборе узла, открытии codemap и source.
- На первой карте: 1 заголовок проекта и 4–8 крупных архитектурных областей.
- Карточка области содержит только: название, одну короткую строку смысла и визуальный тип/акцент. Не показывать длинные prompt summaries и отладочные поля.
- Связь на карте означает одно конкретное правило: **модуль/область A импортирует или использует B**. Направление стрелки и стиль одинаковы всюду.
- Карта должна быть читаема без zoom: нет наложений карточек, нет обрезанных ключевых заголовков, нет «лапши» из рёбер.
- Pan, zoom, selection и minimap работают.

### 3. Правая часть — детали выбранной области

- Пока узел не выбран — лаконичный empty state: «Выберите область на карте».
- При выборе области — title, 1–3 предложения summary, список связанных файлов.
- Вкладка/действие **Codemap** показывает две понятные секции: «Использует» и «Используется в», плюс связанные locations.
- Click location открывает исходник на RIGHT и подсвечивает строку.
- Back/Esc возвращает по понятной цепочке: Source → Codemap → Content → empty.

### 4. Генерация и доверие

- LLM может предложить названия областей и summary.
- Файлы, связи, anchors и layout проверяются/строятся локально, насколько это возможно.
- Если LLM недоступна или её ответ плохой, пользователь всё равно получает простую карту по файловой структуре и import graph; не экран ошибки и не случайную пустую диаграмму.
- В UI видно, что карта построена из cache или пересоздана, но без технического шума.

## Негативный scope MVP

Не включать в текущую задачу:

- редактирование кода, IDE-функции, терминал;
- symbol-level graph для всего репозитория;
- совместную работу, аккаунты, облако;
- множество режимов диаграмм;
- настройку модели в UI;
- pixel-perfect копирование чужого UI;
- упаковку, подписывание и release automation, если они не нужны для ручного smoke.

## Правила постановки задач агенту

Каждая задача должна описывать **наблюдаемый результат**, а не желаемую внутреннюю реализацию.

Плохая формулировка:

> Подключи `DepGraph` к projector и перепиши layout.

Хорошая формулировка:

> После Open Folder на fixture карта содержит project header, 4–8 читаемых областей; между областями видны только реальные import/use связи; карточки не пересекаются; click области открывает минимум два существующих файла на RIGHT.

В задаче обязательно указывать:

1. **Состояние входа:** какой fixture/demo открыть.
2. **Состояние выхода:** что видно на LEFT и RIGHT.
3. **Ограничения:** что не трогать.
4. **Доказательство:** screenshot, unit test или ручной smoke-скрипт.
5. **Один критерий «не принято»:** например, «есть висячие edges», «source не открывается», «карточки наложились».

## План доведения MVP

### Этап A — зафиксировать видимый результат

**Цель:** перестать спорить о слове «красивая карта».

- Выбрать один маленький fixture и текущий репозиторий как два контрольных сценария.
- Для fixture вручную создать `docs/golden-semantic-map.canvas`: header, 4–6 областей, реальные anchors и связи.
- Сделать screenshot ожидаемого состояния рядом с `proto1.png`.
- Составить короткий checklist визуального соответствия: split, toolbar, card hierarchy, edge density, right pane.

**Готово, когда:** любой человек может открыть fixture и за минуту сравнить результат с golden screenshot без знания кода.

### Этап B — довести shell и canvas до референса

**Цель:** интерфейс выглядит и читается как `proto1`.

- Убрать из основного интерфейса phase/debug labels и случайные emoji.
- Привести цвета, размеры карточек, типографику, рёбра, grid, selected state и spacing к одному набору токенов.
- Дать разный визуальный вес header, area и file/location nodes.
- Не менять генерацию семантики на этом этапе.

**Готово, когда:** golden canvas визуально соответствует `proto1`, а LEFT/RIGHT interaction сохранён.

### Этап C — минимально надёжная initial map

**Цель:** Open Folder стабильно даёт полезную карту, не обещая «идеальное понимание кода».

- Вход: file index, package/entry files, file-level imports, опционально LLM areas/summaries.
- Выход: 4–8 areas, существующие file anchors, только валидные edges, детерминированный layout.
- При плохом LLM-ответе — fallback из folder/import structure.
- Проверять: anchors существуют; edge endpoints существуют; нет overlap; у каждого area есть название и минимум один файл.

**Готово, когда:** карта fixture и текущего репозитория читаема после Regenerate и не деградирует при отсутствии LLM.

### Этап D — полезный drill-down

**Цель:** каждая карточка на карте отвечает на вопрос «что это и с чем связано?».

- Content: краткий смысл + files.
- Codemap: Uses / Used by + locations.
- Source: открытие файла и строки; Back/Esc.
- Click file/dep либо открывает source, либо выбирает существующий узел на LEFT — не делает ничего молча.

**Готово, когда:** полный главный сценарий работает на обоих контрольных проектах.

### Этап E — polish только после smoke

**Цель:** убрать friction, не расширяя продукт.

- Починить overflow, пустые состояния, ошибки, loading, cache status.
- Добавить поиск узла только если карта уже читаема без него.
- Сделать 3–5 screenshot-ов реальных состояний: empty, map, selected, codemap, source.

**Готово, когда:** пользователь может показать продукт без объяснения его внутренностей.

## Приёмка MVP

Минимум для «готово к показу»:

- [ ] Open Folder → карта отображается без ручной настройки.
- [ ] LEFT canvas не исчезает при любых режимах RIGHT.
- [ ] 4–8 понятных area cards; без overlap и висячих edges.
- [ ] Любая area открывает Content с существующими file anchors.
- [ ] Codemap показывает реальные связи или честное «связи не найдены».
- [ ] Source открывается минимум для одного anchor каждого area.
- [ ] Карта и shell визуально сверены с `proto1.png` по golden fixture.
- [ ] Без LLM есть fallback-map.
- [ ] Пройдены typecheck, unit tests и ручной smoke на fixture.

## Формат следующего prompt-а агенту

```text
Работай только над этапом <B|C|D> из docs/PRODUCT_BRIEF_MVP.md.

Контрольный сценарий: <fixture / текущий repo>.
Наблюдаемый результат: <одно проверяемое предложение>.
Референс дизайна: docs/proto1.png.
Не менять: <список вне scope>.
Доказательство готовности: <screenshot + тест/ручные шаги>.

Не начинай следующий этап. Если результат требует нового продуктового решения,
остановись и опиши варианты вместо того, чтобы выбирать молча.
```
